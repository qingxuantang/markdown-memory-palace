# Memory Palace — Architecture

This document explains how the palace is organized and **why** each design choice exists. Read it once before doing serious work; refer back when you're unsure how to handle an edge case.

## Mental model

The palace is a **hierarchical knowledge container** with five nested layers:

```
Palace
└── Wing                  # top-level subject domain (e.g. "infrastructure")
    └── Room              # specific topic within a wing (e.g. "web-api")
        └── Hall          # one of 5 fixed memory categories
            ├── Bullet    # short atomic fact (< 300 chars)
            ├── Section   # medium-length entry (300-800 chars, in-file H2)
            └── Drawer    # long-form verbatim entry (own file, with frontmatter)
```

Plus three lateral concepts:

- **Closet** — a per-wing or per-room summary (`_closet.md`), generated by the `closet` command
- **Tunnel** — a bidirectional cross-wing link between two related rooms
- **Audit** — a contradiction report stored under `audits/`, generated by the `check` command

## Why this structure

The MemPalace project's research showed that **structural organization alone improves memory retrieval by ~34%** vs flat semantic search. The intuition: an LLM with a hierarchical map can navigate to the right neighborhood first, then search semantically inside it. Compare:

- **Flat semantic**: "find anything related to 'permission'" → searches all 200 memories, 80% noise
- **Hierarchical**: "go to infrastructure → web-api → discoveries → permission" → searches 5 memories, all relevant

Five **halls** per room is the magic number from MemPalace's research. Fewer collapses too many distinct memory types together; more makes routing decisions ambiguous.

## File-system mapping

Concept → file system:

| Concept | File / dir |
|---|---|
| Palace | `${PALACE_ROOT}/` (= `~/.claude/projects/<cwd-slug>/memory/`) |
| Wing | `${PALACE_ROOT}/wings/<wing>/` |
| Room | `${PALACE_ROOT}/wings/<wing>/<room>/` |
| Hall | `${PALACE_ROOT}/wings/<wing>/<room>/<hall>.md` (one of facts/events/discoveries/preferences/advice) |
| Drawer | `${PALACE_ROOT}/wings/<wing>/<room>/drawers/YYYY-server-c-DD-<slug>.md` |
| Closet | `${PALACE_ROOT}/wings/<wing>/_closet.md` (wing) or `<room>/_closet.md` (room) |
| Tunnel | A markdown link in the `## Tunnels` section of two `_closet.md` files |
| Audit | `${PALACE_ROOT}/audits/YYYY-server-c-DD-<scope>.md` |

## Two-layer memory loading

The palace explicitly distinguishes:

1. **L1 wake-up layer** — four files at `${PALACE_ROOT}/` root: `user.md`, `project.md`, `reference.md`, `feedback.md`. Claude Code auto-loads these at session start. They are **indices**, not content. Each one lists wings and provides ~3 lines per wing as orientation. Total target: < 5000 tokens combined.

2. **L2 deep-dive layer** — everything under `wings/`. Claude reads these on-demand using Read/Glob/Grep when a task requires it. There is no token budget here; rooms can be arbitrarily large.

This mirrors MemPalace's L0/L1 wake-up philosophy: load minimal context at start, fetch detail on demand.

### Why four index files?

Claude Code has a convention of "typed memory" — `user.md` (about the user), `project.md` (about the current project), `reference.md` (reference material), `feedback.md` (recent corrections). The palace maps wings to these indices like so:

| Wing | Indexed in |
|---|---|
| `personal` | `user.md` |
| `projects` | `project.md` |
| `infrastructure` | `reference.md` |
| `people` | `user.md` (collaborators are part of "the user's world") |
| `knowledge` | `reference.md` |
| Other wings | `reference.md` (default) |
| Recent corrections | `feedback.md` always |

The routing prompt knows this mapping. When a new wing is created, the appropriate index gets a new entry.

## Why YAML frontmatter on drawers

Frontmatter fields enable:
- **Time queries** without a database (`valid_from` / `valid_until` + Grep)
- **Provenance** (`source`: was this written by Claude Code, another AI client, the user manually, or imported?)
- **Tagging** for cross-cutting search
- **Supersession** chains (`superseded_by` points to a newer drawer; the old drawer is kept for history)
- **Cross-references** (`related: [path1, path2]`)

Hall files (facts.md, events.md etc.) do NOT have frontmatter — they're append-only logs of bullets and sections. The structure is encoded in the file's directory location.

## Why halls are file-per-category, not tags-on-bullets

Two reasons:

1. **Atomic reads** — when Claude wants "all the facts about web-api", reading one file is faster than scanning a tagged log
2. **Natural append discipline** — having a separate `events.md` makes it obvious that an event needs a date prefix; mixing facts and events in one file would muddy the conventions

The trade-off: a single memory that's both a fact and an event needs to be split, or you pick the dominant category. The `halls-guide.md` document has tiebreaker rules.

## Why no embeddings or vector DB

Three reasons:

1. **Lock-in avoidance** — embeddings tie you to a specific model and vector store; markdown ties you to nothing
2. **Claude is the search engine** — Claude's reasoning over a Grep-narrowed file set outperforms naive vector search for small corpora (< 10,000 memories)
3. **Cost** — vector indexes need maintenance, re-embedding on model upgrades, etc. Alexdown + Grep is free forever

For very large palaces (> 100k drawers) this trade-off changes. At that scale, consider exporting to a real memory system. Until then, the simplicity wins.

## Why git is the concurrency model

Two AI clients (e.g. Claude Code + another AI client) reading the same `${PALACE_ROOT}` is fine — concurrent reads are safe. **Concurrent writes are NOT directly safe** (same as any filesystem), but git provides:

- Per-commit atomicity
- Merge conflict detection at the line level
- Full history (`git log`)
- Provenance (`git blame`)
- Easy rollback (`git revert`)

Convention: put the palace under git. When two clients both want to write, each commits its change; conflicts surface as normal git conflicts and are resolved by the user. This is **not** real-time sync — it's eventual consistency with human-mediated merges. For a single-user, single-machine setup this is more than enough.

## When to break these rules

Some places where the rules can bend:

- **One-off projects** — if you're prototyping and don't need long-term retention, skip the palace entirely; use a `CLAUDE.md` file at the project root
- **Throwaway content** — debug logs, command outputs etc. don't belong in the palace; use `/tmp` or a scratch directory
- **Sensitive data** — secrets never go in the palace, even with frontmatter restrictions; use a real secrets manager

## Glossary

- **L0** — the absolute minimum context loaded at session start (the user's `~/CLAUDE.md` file). Identity layer.
- **L1** — the four index files at the palace root. Topology layer.
- **L2** — the contents of `wings/`. Detail layer.
- **Bullet** — a short append-only entry in a hall file
- **Section** — a medium-length entry as a `## H2` block in a hall file
- **Drawer** — a long-form entry as its own file under `drawers/`
- **Closet** — a generated summary of a wing or room
- **Tunnel** — a cross-wing link between two related rooms
- **Hall** — one of {facts, events, discoveries, preferences, advice}
- **Wake-up** — the act of (re)loading L1 index files into Claude's context
